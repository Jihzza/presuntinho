🐷 Perfil v2 — plano completo

Transformar a página de perfil do Presuntinho de um cartão básico com posts de texto num perfil social a sério — inspirado no melhor do Twitter/X, Instagram, YouTube e TikTok — com publicações de fotos, vídeos, áudio e ficheiros.

SvelteKit 2 + Svelte 5 Supabase (RLS + Storage) PWA local-first i18n ×5 línguas

1. Auditoria do estado atual

O perfil vive em /u/?h=<handle> (src/routes/u/+page.svelte, 586 linhas) com a camada de dados em src/lib/social/posts.ts e a migração 0016_posts_and_avatars.sql. O /perfil/ apenas redireciona.

O que já existe ✅

  • Identidade: avatar (foto webp 512px ou emoji), nome, @handle, bio (160 chars).
  • Capa: gradiente fixo gerado por CSS — não personalizável.
  • Relações: máquina de estados amigo/casal com UMA ação clara por estado (muito bom, mantém-se).
  • Posts v1: só texto (1–500 chars), likes, comentários (300 chars), apagar. RLS: autor + amigos aceites.
  • Stats: apenas nº de publicações (+ badge amigos/casal).
  • Partilha: Web Share API / clipboard.

O que está em falta ❌

  • Publicar qualquer tipo de media: imagens (multi), vídeo, áudio, documentos/ficheiros.
  • Capa/banner personalizada; campos de perfil (site, localização, "membro desde").
  • Tabs de conteúdo (tudo numa lista única); grelha de media; visualizador (lightbox).
  • Post fixado, edição de post (com marca "editado"), contagem de amigos e de gostos recebidos.
  • Ligações clicáveis na bio/posts.

2. Estudo das plataformas

Anatomia dos perfis das quatro referências e o que cada uma ensina para o nosso caso (app íntima, mobile-first, entre amigos).

🐦 Twitter / X

  • Banner 3:1 + avatar a cavalo na capa (já imitamos o layout).
  • Linha meta: 📍 localização · 🔗 site · 📅 "Joined Mês Ano".
  • Stats inline: following / followers.
  • Tabs: Posts · Replies · Media · Likes.
  • Post fixado no topo; grelha de imagens 1–4 com cantos partilhados; badge "edited".

Lição: o modelo base do nosso perfil já é este — falta a media grid 1–4, o pin, a linha meta e as tabs.

📸 Instagram

  • Stats em tripla: posts · followers · following por cima da bio.
  • Grelha 3×N de media como vista principal; ícone ▶ sobre vídeos e 🖼 sobre carrosséis.
  • Carrossel multi-imagem num post; lightbox a ecrã inteiro.
  • Link na bio como campo de primeira classe.

Lição: a tab "Media" em grelha quadrada é a forma mais rápida de rever memórias visuais — perfeita para um casal.

▶️ YouTube

  • Banner largo + avatar + handle + descrição curta com "mais".
  • Tabs por tipo de conteúdo (Videos, Shorts, Playlists, About).
  • Secção "About": descrição longa, links, data de adesão, contagens.
  • Player inline com poster antes de reproduzir.

Lição: vídeo merece player nativo inline (poster + controls), não um link; "membro desde" dá história ao perfil.

🎵 TikTok

  • Stats em tripla: following · followers · likes recebidos (soma de todos os gostos).
  • Grelha vertical de vídeos com pré-visualização; posts fixados marcados "Pinned".
  • Perfil extremamente mobile-first, ações grandes.

Lição: "likes recebidos" é uma métrica calorosa e gamificada — encaixa no espírito do Presuntinho. Pin também aqui.

4. Especificação técnica

4.1 Base de dados — migração 0017_profile_v2_media.sql

-- accounts: campos de perfil novos
alter table public.accounts add column if not exists cover_url text;
alter table public.accounts add column if not exists website   text;
alter table public.accounts add column if not exists location  text;

-- posts: media (jsonb [] de anexos), pin único por autor, marca de edição
alter table public.posts add column if not exists media jsonb not null default '[]'::jsonb;
alter table public.posts add column if not exists pinned boolean not null default false;
alter table public.posts add column if not exists edited_at timestamptz;
-- body passa a poder ser vazio QUANDO há media
alter table public.posts drop constraint posts_body_check;
alter table public.posts add constraint posts_body_check check (
  char_length(body) <= 500 and (char_length(body) > 0 or jsonb_array_length(media) > 0)
);
create unique index posts_one_pin_per_author on public.posts (author) where pinned;

-- bucket PRIVADO post-media/<author-uuid>/<post-media-id>
-- leitura: can_see_posts(auth.uid(), pasta) → amigos aceites; escrita: só a própria pasta

Cada elemento de posts.media: { kind: 'image'|'video'|'audio'|'file', path, mime, name, size, width?, height? } — o path aponta para o bucket privado; o cliente pede signed URLs em lote (TTL 1 h), como o chat já faz.

4.3 Camada cliente — src/lib/social/posts.ts

  • createPost(body, files) — valida combinações (imagens ⊕ 1 vídeo ⊕ 1 áudio ⊕ ficheiros), comprime, faz upload e insere.
  • resolveMediaUrls(posts) — createSignedUrls em lote com cache em memória.
  • setPinned(post, on) — desafixa o anterior (índice único garante 1).
  • updatePost(id, body) — grava edited_at.
  • countFriends(), countLikesReceived(authorId) — para a stats tripla.
  • linkify(text) — só https://, escapado, para bio e posts.

4.6 i18n

Todas as strings novas entram como $t('key', { default }) e são adicionadas às 5 línguas (pt-PT, en, fr, ar, tn) com paridade exata de chaves; ar herda o RTL global e tn fica em escrita latina.

6. Riscos e decisões

  • Privacidade da media: bucket privado + RLS can_see_posts na pasta do autor; URLs assinados de curta duração. Nada de bucket público para conteúdo de amigos.
  • Sem transcode de vídeo no browser: aceitamos o ficheiro tal como vem (mp4/webm/mov) com limite de 50 MB — igual ao compromisso do chat.
  • Migração aditiva: só add column/novo índice/novo bucket — zero impacto nos dados existentes; o novo check do body é mais permissivo que o antigo (posts antigos continuam válidos).
  • Compatibilidade: clientes antigos ignoram media (leem só body); posts sem texto aparecem vazios até atualizarem — aceitável numa app de 2 utilizadores.
  • Órfãos no storage: apagar post apaga a linha; os objetos ficam órfãos (sem acesso além do autor). Limpeza best-effort no cliente ao apagar.
